# Provenance Guard

A backend system that classifies submitted creative content, scores confidence, surfaces transparency labels to users, and handles appeals from creators who believe they have been misclassified.

Built for CodePath AI201 Week 4.

---

## Architecture Overview

Provenance Guard is a Flask REST API that processes text submissions through two independent detection signals, combines their outputs into a unified confidence score, and returns a human readable transparency label.

```
Client
  |
  v
POST /submit
  |
  +---> Signal 1: LLM Classification via Groq
  |           Prompts the model to estimate AI generation probability
  |
  +---> Signal 2: Stylometric Heuristics
              Measures sentence variance, vocab richness, punctuation density
  |
  v
Confidence Scorer  ->  Weighted average -> score 0.0 to 1.0
  |
  v
Transparency Label  ->  One of three plain language labels
  |
  v
Audit Logger  ->  Structured JSON log entry written
  |
  v
JSON Response returned to client
```

## Detection Signals

### Signal 1: LLM Classification via Groq

**What it measures:** The model reads the submitted text and estimates the probability that it was generated by an AI system rather than a human.

**Why chosen:** Large language models have strong pattern recognition for their own outputs. They detect subtle regularities in structure, hedging language, and sentence construction that are invisible to rule based systems.

**Limitations:** The model may be overconfident on short texts. It may also struggle with content that was heavily edited after AI generation, or with creative writing styles that intentionally mimic formal AI output.

### Signal 2: Stylometric Heuristics

**What it measures:** Statistical features extracted from the raw text including average sentence length, standard deviation of sentence length, type to token ratio for vocabulary richness, comma density per 100 words, and frequency of hedging phrases like "it is important to note" or "in conclusion".

**Why chosen:** Human writing tends to have higher variance in sentence length, more irregular punctuation, and a more personal vocabulary. AI generated text is statistically more uniform. This signal requires no external API call, making it fast and always available as a fallback.

**Limitations:** Polished academic writing by humans can score similarly to AI output. Texts under 50 words produce unreliable variance estimates.

## Confidence Scoring

The final confidence score represents how confident the system is that the content is AI generated, on a scale from 0.0 (certainly human) to 1.0 (certainly AI).

```
final_score = (llm_score * 0.65) + (stylometric_score * 0.35)
```

The LLM signal is weighted higher because it captures semantic patterns that heuristics miss. The stylometric signal acts as a grounding check.

### Example A: High AI Confidence

**Input:** "Artificial intelligence represents a transformative paradigm shift in modern society. It is important to note that while the benefits of AI are numerous, it is equally essential to consider the ethical implications. Furthermore, stakeholders across various sectors must collaborate to ensure responsible deployment."

| Signal | Score |
|---|---|
| LLM Classification | 0.85 |
| Stylometric | 0.48 |
| **Final Score** | **0.72** |

When the LLM signal is available, this text scores in the high-confidence AI range. The stylometric signal detects low sentence variance and hedging phrases.

### Example B: High Human Confidence

**Input:** "ok so i finally tried that new ramen place downtown and honestly? underwhelming. the broth was fine but they put WAY too much sodium in it and i was thirsty for like three hours after. my friend got the spicy version and said it was better. probably wont go back unless someone drags me there"

| Signal | Score |
|---|---|
| LLM Classification | 0.08 |
| Stylometric | 0.16 |
| **Final Score** | **0.11** |

**Label:** Human Authored (High Confidence)

The stylometric signal scores very low (0.16) for this text due to high sentence length variance, informal vocabulary, and zero hedging phrases — strong human indicators.

## Transparency Labels

All API responses include a plain language label. There are three variants.

**AI Generated (High Confidence)** triggered when final score >= 0.75

Message: "Our system found strong indicators that this content was generated by an AI tool. This label is here to help readers understand how the content was made."

**Human Authored (High Confidence)** triggered when final score <= 0.30

Message: "Our system found strong indicators that this content was written by a human. No AI generation signals were detected."

**Uncertain Origin** triggered when final score is between 0.31 and 0.74

Message: "Our system could not determine with confidence whether this content was written by a human or generated by AI. If you believe this classification is wrong, you can submit an appeal."

## API Endpoints

### POST /submit

Accepts a text submission and returns the attribution result. Rate limited to 10 requests per minute per IP.

**Request:**
```json
{
  "text": "Your text here",
  "creator_id": "user_abc123"
}
```

**Response:**
```json
{
  "content_id": "uuid",
  "confidence": 0.88,
  "label": "AI Generated (High Confidence)",
  "label_message": "Our system found strong indicators...",
  "signals": { "llm": 0.91, "stylometric": 0.82 }
}
```

### POST /appeal

Allows a creator to dispute their classification. Sets status to under review.

**Request:**
```json
{
  "content_id": "uuid",
  "creator_reasoning": "I wrote this myself. I use formal language professionally."
}
```

**Response:**
```json
{
  "content_id": "uuid",
  "status": "under_review",
  "message": "Your appeal has been received and will be reviewed."
}
```

### GET /log

Returns a structured JSON audit log of all decisions and appeal statuses.

**Response:**
```json
[
  {
    "submission_id": "uuid",
    "timestamp": "2026-06-28T23:00:00Z",
    "score": 0.88,
    "label": "AI Generated (High Confidence)",
    "appeal_status": null
  }
]
```

## Rate Limiting

| Endpoint | Limit | Reasoning |
|---|---|---|
| POST /submit | 10 per minute; 100 per day | A legitimate creator submits at most a few pieces per session. 10/min stops automated scrapers while leaving headroom for normal use. 100/day caps persistent abuse. |
| POST /appeal | 5 per hour per IP | Appeals are low frequency by nature; a tight limit prevents spam flooding the review queue |
| GET /log | 30 per minute per IP | Read only endpoint; higher limit allows monitoring tools to poll without interruption |

## Known Limitations

- Texts under 50 words produce low confidence scores regardless of origin because there is not enough signal for either detector.
- The LLM signal costs Groq API tokens on every submission. High traffic will drain credits quickly without a caching layer.
- Stylometric features were calibrated on English language text. Non English submissions will produce unreliable scores.
- The appeals workflow logs appeals and marks them as under review but does not implement a human review interface. That step remains manual.
- Detection patterns reflect AI writing styles as of mid 2026. Newer models may produce text that evades current signals.

---

## Spec Reflection

The hardest design decision was choosing the weighting between the two signals. A pure LLM approach offered no explainability. A pure heuristics approach was too brittle on edge cases. The 65/35 split was chosen after testing both extremes on a small set of known human and AI samples.

The appeals workflow surfaced a real gap: the spec asks to log appeals and set status to under review, but in a real system someone has to actually review them. I noted this as a known limitation rather than pretending it is solved.

## AI Tool Usage

### Instance 1: Full Implementation Generation

**Directed:** Asked Claude to generate the complete Flask application (app.py, detector.py, scorer.py, audit.py) based on the planning.md spec — providing the architecture diagram, signal descriptions, confidence thresholds, and label variants as context.

**Produced:** Working Flask app with all routes, Groq integration, stylometric analysis, confidence scoring, and JSON audit logging.

**Revised:** Adjusted the stylometric feature weights after testing showed the initial hedging-phrase signal was too dominant. Also added the `word_count < 50` short-text handling that the spec called for but the generated code initially missed.

### Instance 2: Debugging Stylometric Scoring

**Directed:** Asked Claude to review why the stylometric signal was returning near-identical scores for clearly different texts.

**Produced:** Identified that the type-token ratio calculation was inverted (higher TTR was being scored as more AI-like when it should be the opposite for diverse vocabulary).

**Revised:** Flipped the TTR scoring direction so that lower vocabulary diversity (more repetitive text) scores higher on the AI-likelihood scale.
